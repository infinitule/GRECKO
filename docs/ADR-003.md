# ADR-003: Rules-first classifier, learned-later interface

**Status:** Accepted  
**Phase:** P3 — Classification (transparent, swappable)

## Context

Classification sits between the fusion air picture and the allocator. Its outputs
(label, confidence, priority score, rationale) are consumed directly by the human
operator in the C2 console and by the allocator's value estimate. Two tensions
drive the design:

1. **Explainability requirement:** a human-on-the-loop system must be able to
   audit why any track was labelled hostile and why any engagement was authorised.
   A neural classifier that outputs a probability but no rationale is insufficient.
2. **Upgradeability:** the v1 ruleset will be replaced by a learned model once
   the league (Pillar C) generates sufficient training data. The classifier must
   be swappable behind a stable interface.

## Decisions

### `Classifier` interface: `classify(features, is_known_friendly, asset_pos, track_state) -> ThreatAssessment`

All callers depend on this interface only. `RuleClassifier` is the v1
implementation; a learned model implements the same method. The pipeline holds
an injected `Classifier` instance — swap at construction time, no other code
changes.

### `ThreatAssessment.why` is required and non-empty

The `why` field is a required string on the output dataclass and on the
`/proto/threat_assessment.schema.json` contract. The acceptance test asserts
it is non-empty on every assessment. This is intentional architecture:
explainability is enforced structurally, not by convention.

### Rule scoring: weighted boolean features -> normalised confidence

Five rules (approach_rate, heading_to_asset, speed_above_bird, low_weave_energy,
track_maturity) contribute weighted scores. Confidence = weighted mean; rationale
string lists which rules fired and which did not. This is fully auditable and
mirrors the human operator's own mental model of the threat.

Weights were set to make the dominant term time-to-asset (approach_rate weight=3,
heading weight=2.5) while bird-rejection heuristics carry lighter weight.

### Friendly protection: three-rule corroboration gate

A `mark_friendly` flag caps any track at "unknown" UNLESS all three primary
hostile indicators (approach_rate, heading_to_asset, speed_above_bird) fire
simultaneously. "All three" is the most restrictive gate consistent with the
requirement that a friendly aircraft *actively executing a kamikaze ingress* can
still be escalated. The 1000-trial stress test confirms zero violations.

### Priority score: `(1 / TTA) × confidence × hostile_multiplier`

Time-to-asset is the dominant term as specified. Multiplied by confidence so an
uncertain threat prioritises lower than a certain one at the same range.
`hostile_multiplier` (1.0/0.5/0.1 for hostile/unknown/friendly) prevents the
allocator from bidding heavily on labelled-friendly tracks.

### Track history window

20 TrackMessages per track are stored for weave-energy computation. This is the
only statefulness in the pipeline; everything else is a pure function of the
current state.

## Rejected alternatives

- **Learned classifier from the start:** no training data exists until Pillar B
  generates scripted doctrines and the league runs. Rules-first is the only
  honest choice for v1. The interface keeps the path open.
- **Probability-only output without rationale:** fails the human-on-the-loop
  audit requirement. Rejected.
- **Separate "classify" and "prioritise" stages:** priority score is a function
  of classification and track state; combining them into one `ThreatAssessment`
  message keeps the bus simple and avoids partial-update races. Rejected.
- **Harder corroboration gate (all five rules):** mature-track and low-weave-energy
  rules can both be absent for a real threat (new fast-moving target with no weave);
  using only the three kinematics-primary rules avoids false negatives. Rejected.

## Acceptance evidence

- 1000 random birds: non-hostile rate ≥ 95% (actual: tracked in test).
- Priority ordering: closer/faster inbound tracks rank higher (verified).
- Every assessment: `why` non-empty, contains rule-fire list and conf/tta values.
- Friendly protection: 1000 trials, zero hostile labels without full corroboration.
- Stub classifier satisfies the interface, confirming swappability.
