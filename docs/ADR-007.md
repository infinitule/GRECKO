# ADR-007: Effector taxonomy and type-aware allocation

**Status:** Accepted  
**Phase:** PE — Effector heterogeneity (parameters only)

## Context

A real counter-UAS deployment deploys a mixed fleet: expensive kinetic
interceptors, cheap net-capture drones, EW-effect payloads that work only on
RF-dependent threats, and low-cost collision drones. A homogeneous "one type
of interceptor" assumption makes the cost-exchange optimisation trivially easy
and misses the portfolio problem that is most of Pillar A's value. The allocator
must choose BOTH which target to engage AND which effector type to assign.

## Scope boundary

This module is parameter sets only. The EW entry (`ew_soft_kill`) is explicitly
described as "a kill-probability parameter for the sim allocator" — not a jammer,
not a waveform, not a DRFM model. The description field, enforced by a test,
carries this disclaimer. This is the code-level enforcement of the project's
scope boundary for this type.

## Decisions

### Four canonical effector types

| Type | Cost | Max Pk quadrotor | Max Pk RF-dep | Notes |
|---|---|---|---|---|
| kinetic_interceptor | $90k | 0.85 | 0.80 | Highest Pk, highest cost; last resort |
| net_capture_drone | $3k | 0.78 | 0.60 | Cheap, non-lethal; slow (22 m/s) |
| ew_soft_kill | $500 | 0.00 | 0.90 | Soft-kill only; invalid vs non-RF threats |
| collision_drone | $800 | 0.70 | 0.65 | Single-use; faster than net; one-shot |

### `p_k_table` keyed by `threat_type` string

The threat_type key is the bridge between classification (P3) and allocation (PA).
`RuleClassifier` currently does not emit a `threat_type`; PA will default to
`"quadrotor"` until P3 is extended with a type field. The interface is already
correct: `EffectorType.p_k(threat_type)` returns the table value or falls back
to the minimum (conservative). No code changes needed to wire in a typed
classifier.

### `geometry_valid(range_m, aspect_angle_rad)` as the engagement feasibility gate

The allocator calls this before computing expected kill value. An infeasible
engagement (out of range, wrong aspect) is excluded from the assignment matrix
entirely. This keeps the cost-exchange computation from assigning a net drone to
a target 2 km away.

### `best_effector_for(threat_type)` returns cost-adjusted max Pk = Pk / cost

This is the single-step decision rule when the allocator has no magazine budget
constraint — the greedy baseline uses it directly. The EconomicMDP uses the
full objective over the rolling horizon instead.

## Rejected alternatives

- **Continuous effector parameters (caliber, seeker type, etc.):** adds fidelity
  that is indistinguishable from RF/fire-control design at the margin. Four
  discrete types + Pk table is the right abstraction level for this sim. Rejected.
- **Single effector type with cost as a knob:** loses the heterogeneous assignment
  problem. The interesting investor demo (EW clears RF-dependent wave cheaply;
  kinetic reserved for hardened targets) requires type diversity. Rejected.
- **Effector parameters embedded in the world YAML:** the catalogue is shared by
  all scenarios; duplicating it would immediately diverge. Central catalogue
  with YAML scenarios referencing by `effector_id` is cleaner. Rejected.
