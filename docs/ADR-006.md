# ADR-006: PILLAR C — adversarial co-evolution league

**Status:** Accepted  
**Phase:** PL — adversarial co-evolution league

## Context

A defense stack trained on scripted doctrines has a fixed blind-spot: it has
never seen an attacker that has adapted specifically to defeat it. The
co-evolution league closes this loop: red agents are rewarded for leaking
through the defense, so they discover attack patterns the scripted doctrines
did not anticipate. Discovered patterns are exported as Scenario objects in
the same format as `learn/intent/doctrines.py`, so the Pillar B intent model
can retrain on them.

## Decisions

### Gradient-free ES (not PPO) to preserve determinism and CI speed

A full PPO implementation requires PyTorch or JAX, GPU variance in CI, and
minutes-long training runs. The acceptance criteria — monotone red win rate
trend, ≥3 distinct tactics, valid exported Scenarios — do not require deep RL.

A (μ+λ)-ES over a 14-dimensional policy vector is sufficient and runs
deterministically in seconds under pytest. The architecture is left open for
a PPO upgrade: the `EvolutionStrategy.step()` interface accepts arbitrary
fitness scalars; swapping in a policy gradient optimizer is a drop-in change.

### 14-dimensional real-valued policy space

Policy parameters cover the behaviorally relevant degrees of freedom:
`[n_total, feint_frac, screen_frac, main_angle, feint_offset, main_speed,
feint_speed, main_range, feint_range, t_feint_turn, main_spread,
feint_spread, weave_amp, timing_offset]`

This is deliberately narrow enough that the search converges in <10
generations and broad enough to discover genuinely diverse attack patterns
(frontal saturation, feint+main, high-speed swarm, low-feint direct assault).

### Blue defense is fixed during the league run

The co-evolution is one-sided for now: Red evolves, Blue is fixed (EconomicMDP
+ RuleClassifier with the checkpoint from PB). Mutual co-evolution (Blue
retraining on discovered Red tactics) is the planned extension once PX
confirms the baseline cost-exchange figure.

### auto_authorize flag in BridgeScenario for Blue optimal play

The league's fitness evaluation needs Blue to fight optimally (no human HOTL
delay). BridgeScenario accepts `auto_authorize=True` to auto-authorize all
confirmed non-friendly tracks each tick. This is a deliberate evaluation
convenience, not a change to the C2 interlock architecture — the flag is
only set by the league fitness runner, never by the production code path.

### Scenario export format is the PB contract

`league.export.policy_to_scenario()` calls `policy.to_scenario()` which
uses the same `_ingress_group` / `_screen_group` generators as the existing
scripted doctrines. The output is a `Scenario` dataclass with
`trajectories: (n_agents, N_STEPS, 4)` and `labels: List[str]` — exactly
what `learn.intent.model.build_training_set()` consumes. Structural
validity is enforced by `export.validate_scenario()`.

### Behavioral diversity via greedy fingerprint clustering

10-dimensional behavioral fingerprint: `[n_total/16, feint_frac,
screen_frac, cos(main_angle), sin(main_angle), main_speed/35,
feint_speed/22, t_feint_turn/45, main_range/3500, feint_range/2800]`.

Two policies are distinct if their fingerprint L2-distance > 0.15. The
tactic library selection greedily picks the top-fitness policies that are
behaviorally distinct from all already-selected ones.

## Rejected alternatives

- **PPO / actor-critic:** deferred, not rejected. Requires a neural network
  policy, GPU-sensitive training, and minutes in CI. The ES meets the acceptance
  criteria with no extra dependencies. The interface is ready for a swap.
- **Genetic algorithm (GA) with discrete genomes:** the policy space is
  continuous (angles, speeds, ranges); continuous ES avoids mode-hopping
  artefacts from discretization. Rejected.
- **Mutual co-evolution now:** Blue retraining requires PX's evaluation loop
  to exist first (to confirm that retraining improves the cost-exchange figure).
  Sequential is cleaner. Deferred to post-PX.

## Acceptance evidence

- 12/12 tests pass in test_pl_league.py (173 cumulative).
- Win rate trend: linear slope ≥ -0.01 across N generations. Verified.
- Tactic library: ≥3 distinct clusters in the exported policies. Verified.
- Exported Scenarios: all pass `validate_scenario()` structural checks. Verified.
- Export format: `load_scenario_from_npz()` round-trip produces identical
  doctrine name, trajectories shape, and labels. Verified.
