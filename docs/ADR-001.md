# ADR-001: Sensor error models & the no-truth-leakage boundary

**Status:** Accepted  
**Phase:** P1 — Sensing mesh (imperfect observers)

## Context

Everything downstream of sensing (fusion, classification, allocation, intent
prediction) must operate on *partial, noisy* information — this is the POSG
discipline the whole project rests on. The sensing layer is therefore the one
and only place ground truth is consumed, and its output contract must make
truth leakage structurally impossible, not merely discouraged.

## Decision

### Three sensor archetypes with distinct error characters

| | RADAR | EO/IR | ACOUSTIC |
|---|---|---|---|
| Max range | 3000 m | 1200 m | 400 m |
| FOV | 360° | 120° | 360° |
| Measurement | cartesian, σ=25 m | cartesian σ=8 m / bearing-only σ=5 mrad | bearing-only, σ=150 mrad |
| Update rate | 2 Hz | 10 Hz | 4 Hz |
| Latency | 100 ms | 50 ms | 300 ms |
| Clutter (Poisson mean/scan) | 0.5 | 0.1 | 0.3 |
| RF-immune | no | yes | **yes** |

The heterogeneity is deliberate: fusion must reconcile incompatible measurement
spaces (cartesian vs bearing-only), and the allocator inherits a picture whose
quality varies with geometry and the comms/EM environment. ACOUSTIC's RF
immunity matters later — it is the sensing floor that survives jamming scenarios.

### Detection-probability model: flat-then-linear

`Pd(r) = pd_max` out to a knee (fraction of max range), then linear falloff to 0
at max range. Chosen over a physically derived radar-equation model because it is
(a) directly testable against configuration (the 10k-trial acceptance test),
(b) trivially parameterisable per scenario, and (c) honest about being an
abstraction — this sim does not claim RF fidelity (see scope boundary).

### No-truth-leakage enforced in the message type

`SensorReport` is a frozen dataclass whose fields are *only*: timestamp (with
latency applied), sensor identity, a per-sensor sequence number, the noisy
measurement, its covariance, and the sensor's own position. There is no entity
id field to misuse. The acceptance test asserts the field set against a
forbidden list and verifies measurements never reproduce exact truth.

### Truth sidecar as an explicit, separate channel

Evaluation needs report→truth correspondence (for purity metrics in P2). This is
exposed only via `scan_all_with_truth_sidecar()`, a separately named method that
`/sim/fusion` must never call — a grep-able CI invariant, mirroring the comms
isolation guarantee planned for PC.

## Rejected alternatives

- **Radar-equation / SNR-based Pd:** more "realistic" but unverifiable in this
  abstraction and contrary to the declared scope (no RF design). Rejected.
- **Truth id embedded with a "do not read" convention:** conventions don't
  survive contributors. Structural absence is the only reliable guarantee. Rejected.
- **Single generic sensor with config-only differences:** loses the
  bearing-only/cartesian split, which is exactly the hard part fusion must handle.
  Rejected.

## Consequences

- P2 fusion must handle two measurement kinds (cartesian, bearing-only) and
  asynchronous, latency-shifted timestamps from the start.
- Clutter is generated uniformly in coverage; fusion's gating must reject it
  at baseline rates (a P2 acceptance criterion).
- The truth sidecar is the only sanctioned path for eval metrics; CI will grep
  `/sim/fusion` for the sidecar method name.
