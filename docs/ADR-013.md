# ADR-013: PD — decentralized swarm-on-swarm coordination

**Status:** Accepted
**Phase:** PD — decentralized coordination (Pillar D)

## Context

Pillars A–C all assume a **central** decision node. The EconomicMDP allocator
(PA) sees every interceptor and every track at once and solves a global
assignment; the C2 bridge (PV) is a single authority; the league (PL/PM) tunes
that central policy. This is optimal when the picture is whole and the C2 link
holds — and brittle exactly where counter-swarm operations are hardest:

- **Single point of failure.** If the link to the central allocator drops, the
  fleet is leaderless.
- **Partial observability of the mesh.** The central solver implicitly assumes
  it can reach every node. The comms layer (PC) already models range collapse,
  regional denial, intermittency, and full partition — but PA only *consumes*
  the resulting adjacency as a hint; it never has to coordinate *through* a
  degraded mesh.

PD adds the decentralized counterpart: every drone is an autonomous agent that
coordinates peer-to-peer over the live comms mesh, on both the defending and
attacking sides. The question PD answers is not "can decentralization beat the
central solver on a clean network" (it cannot — a global optimum dominates a
local one) but **"how much optimality do you trade for surviving comms loss,
and is that trade worth it?"**

**Scope:** simulation and research only. Agents exchange abstract
claim/report messages over the PC topology. Nothing here designs RF, controls
hardware, or computes a launch solution. Every engagement intent PD produces is
an advisory `Assignment`, identical in kind to PA's output, and remains
subordinate to the human-on-the-loop interlock (`C2State.can_engage`). The
INTERLOCK invariant is untouched: PD adds no new `world.assign()` call site.

## Decisions

### The coordination rule is threat-primary, economics-secondary

Stock market-based task allocation (CBBA and relatives) makes the **bid** — a
scalar utility, usually value-for-money — the primary key. GRECKO's central
allocator is likewise economic to its core (`benefit = v·Pk − λ·cost`). PD
deliberately inverts that ordering for the distributed case. When two agents
contest the same track, the winner is decided by, in order:

1. **capability** — `P_k(best feasible effector) × urgency` the agent can bring
   to *this* threat. Who can neutralize it best, and soonest.
2. **cost** — the dollar cost of that effector. The **secondary** tiebreak, used
   only when capability is tied within `1e-6`.
3. **agent id** — a deterministic final tiebreak so replays are byte-identical
   regardless of the order messages happen to arrive.

The rationale: a distributed node acting on a *local* picture should answer the
tactical question first ("can I stop this?") and let cost refine among equals,
rather than let a cheap-but-ineffective option win a bid it cannot honour. The
economic thesis is preserved — cost still decides between equally capable agents,
and the effector menu still spans $800–$90k — but threat response leads. This is
the rule captured in `sim/swarm/consensus.py::resolve`.

### The mesh *is* the negotiation channel, not a hint

PA receives `AllocInput.adjacency` — the connected-components topology PC
publishes each tick — and uses it only to decide which interceptors to solve
jointly. PD uses the *same* structure as the actual message-passing graph.
`DecentralizedDefense` runs synchronous **gossip consensus** over it:

- Each agent assesses its local threat/capability table (`DroneAgent.assess`).
- Each claims its highest-severity winnable threat and broadcasts the claim.
- Every round, agents merge neighbours' beliefs (one hop of propagation per
  round), and any agent whose claim is beaten concedes and re-picks.
- Iteration runs to a fixpoint (no agent changes) or a fleet-size-derived cap.

Information therefore diffuses at exactly the rate the mesh allows. This yields
two provable, tested properties (`sim/tests/test_pd_swarm.py`):

- **Within one connected partition**, consensus converges to a conflict-free
  assignment: no two agents engage the same track (`last_collisions == 0`).
- **Across partitions**, agents that cannot hear each other may both claim a
  commonly-visible threat. That double-commit is **not suppressed** — it is
  counted (`last_collisions`) and emitted as two real `ASSIGN`s, so it shows up
  downstream as extra spend and a worse cost-exchange ratio. Hiding it would
  misrepresent the cost of a partitioned mesh, which is the entire point of the
  study.

### Determinism is preserved by construction

Updates are synchronous (Jacobi-style: every agent decides from the previous
round's state, so the result is independent of agent-visit order). All tiebreaks
bottom out in integer-stable comparisons and agent ids. Optional per-message
loss (`drop_rate`, for comms-stress sweeps) draws from a seeded
`numpy.random.Generator`, so even the lossy path replays byte-for-byte. PD adds
no new nondeterminism to the DETERMINISM invariant's fixed scenario.

### The attacker gets the mirror image

`DecentralizedSwarm` is the Red-side coordinator. The approach is discretised
into angular sectors around the asset; each drone estimates defensive pressure
per sector from the interceptor bearings it can see, shares that belief, and the
swarm water-fills toward the gaps — massing on the least-defended axis and
**re-massing within the episode** as drones are lost or the defence commits. The
heaviest sector becomes the main press; a thin one is tasked as a feint to split
the defence. This composes with PL/PM: the league discovers a strong opening
*between* episodes; the swarm re-solves the approach *live*.

## Consequences

- **A new comparison axis.** `DecentralizedDefense` implements the `Allocator`
  interface, so it is a drop-in third arm alongside EconomicMDP and GreedyMyopic
  in the PX harness. `grecko swarm` runs the head-to-head under a comms-denial
  sweep and reports the optimality-vs-resilience trade directly.
- **Honest degradation, not a magic bullet.** On a clean mesh the central
  allocator wins on raw cost-exchange; PD's value is the *shape of the
  degradation curve* as denial increases — it keeps coordinating (per partition)
  where a central node would go dark.
- **Interlock unchanged.** PD produces advisory assignments only; the single
  guarded `world.assign()` in the bridge is still the sole path into the world.
- **Deferred.** Full kinematic integration of the attacker swarm into
  `BridgeScenario` (so re-massing drives the live world, not just the plan) is
  left to a follow-up; PD ships the coordinator, its tests, and the eval/CLI
  surface. The browser demo visualizes both coordinators end to end.

## Files

- `sim/swarm/messages.py` — `ThreatReport`, `Claim`, `Release` peer messages.
- `sim/swarm/consensus.py` — threat-primary `resolve` / `best_claim`.
- `sim/swarm/agent.py` — `DroneAgent`: local assessment, pick, claim.
- `sim/swarm/defense.py` — `DecentralizedDefense(Allocator)` gossip coordinator.
- `sim/swarm/swarm.py` — `DecentralizedSwarm` attacker-side coordinator.
- `sim/tests/test_pd_swarm.py` — 16-test acceptance suite.
- `eval/decentralized.py` — centralized-vs-decentralized comms-denial study.
- `grecko swarm` — CLI entry point for the study.
